# BDD Scenarios: Attach Document to LC Application (R8.3-UC2) - Reversed

**Requirement:** R8.3-UC2 (CSR attaches necessary documents like Applicant requests or Signed LC scan to the LC record)

This document contains BDD scenarios reversed from the implemented test cases in `TradeFinanceServicesSpec.groovy`.

## Feature: Attach Document to LC Application

**As a** CSR or Trade Operator
**I want to** attach supporting documents to an LC record
**So that** all relevant physical and electronic artifacts are stored centrally for review and audit

### Background
An LC record (Draft or Issued) exists in the system.

### Scenario: Successfully attaching a document (Happy Path)
**Given** the user is viewing an LC record (e.g., `DEMO_LC_07`)
**When** the user calls the **"attach#LcDocument"** service with:
  - **documentTypeEnumId**: "LC_DOC_COMM_INVOICE"
  - **description**: "Test Commercial Invoice"
**Then** the system should create an **LcDocument** record linked to the LC
**And** it should store the document metadata (type, description, and link to file/content)
**And** the system should report a success status.

### Scenario: Verifying automatic MT700 attachment (Happy Path)
**Given** an LC has been successfully issued
**When** the system generates the SWIFT message
**Then** an **LcDocument** of type **"LC_DOC_SWIFT_MSG"** should be automatically created and attached to the LC record.

---

## Business Rules & Validation
1. **Document Types:** The system MUST support various document types (e.g., Application Form, Transport Doc, Invoice, SWIFT Message).
2. **Immutability:** SWIFT messages generated by the system are stored as read-only documents.
3. **Traceability:** Every document MUST be linked to a specific `lcId` and optionally a `drawingSeqId` or `amendmentSeqId` if applicable.
4. **Visibility:** Attached documents must be visible in the "Documents" or "History" tab of the LC detail screen.
5. **Categorization:** Users must select a valid `documentTypeEnumId` from the predefined list to ensure consistency.
